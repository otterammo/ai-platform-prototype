# API

## Purpose

The API exposes platform resources and control-plane operations. API clients use
the API to create desired state, observe status, watch events, approve guarded
actions, and retrieve artifacts and traces.

The API MUST preserve the resource model defined by `ai.platform/v1`.

## REST Conventions

A REST API SHOULD expose resource-oriented endpoints. Resource operations SHOULD
map to create or apply, get, list, watch, delete, and status update semantics.

API paths and transport details MAY vary, but the API MUST preserve resource
identity by kind, name, and scope. Workspace-scoped resources MUST be addressable
within a Workspace namespace.

## Resource Endpoints

The API MUST support submitting one or more resource manifests. It SHOULD
support retrieving a single resource, listing resources by kind and scope, and
deleting resources by identity.

The API SHOULD expose ToolInvocation and Observation resources through the same
resource-oriented semantics as other Workspace-scoped resources. API projections
for AgentRuns and traces SHOULD include ToolInvocation and Observation
references when available.

Apply semantics SHOULD be declarative. Applying a changed spec MUST advance
generation. Applying unchanged desired state SHOULD be idempotent.

## Status Updates

Status is controller-owned. The API MUST distinguish desired-state writes from
status updates. Ordinary resource apply MUST NOT permit clients to overwrite
controller-owned status.

Status updates SHOULD record observedGeneration when the responsible controller
has processed a generation.

## Watch Semantics

The API SHOULD support watch semantics for resources and events. Watch streams
SHOULD deliver enough information for clients to observe lifecycle transitions,
waiting states, failures, and completions.

Watch consumers MUST tolerate reconnects, missed events, and repeated events
unless the platform explicitly offers stronger guarantees.

## Future Streaming

The API MAY support streaming AgentRun logs, partial outputs, tool progress, and
model token events. Streaming output MUST NOT replace Artifact resources,
AgentRun status, or immutable Events.

Streaming channels MUST enforce Workspace, policy, and authorization boundaries.

## Compatibility Guarantees

Compatible v1 APIs MUST preserve:

- Resource envelope fields and meanings.
- Scope and identity semantics.
- Spec and status ownership separation.
- Generation and observedGeneration behavior.
- AgentRun-only execution.
- ToolInvocation and Observation traceability.
- Context as the runtime knowledge boundary.
- Policy and Approval semantics.
- Event traceability.

APIs MAY add optional fields, endpoints, projections, and convenience operations
when they do not weaken these guarantees.

## Errors

API errors SHOULD distinguish validation failure, admission rejection, missing
resource, conflict, forbidden action, policy denial, waiting for approval, and
internal failure.

Error responses SHOULD include machine-readable reason and human-readable
message. Errors SHOULD NOT expose secrets or provider credentials.
