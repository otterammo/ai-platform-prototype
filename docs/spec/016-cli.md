# CLI

## Purpose

The command-line interface provides a resource-oriented operator and developer
experience for the platform. It SHOULD follow the philosophy of `kubectl` where
appropriate: declarative manifests, inspectable resources, clear status, and
composable output.

CLI behavior MUST preserve the API and resource semantics of `ai.platform/v1`.

## Command Philosophy

The CLI SHOULD be a thin client over platform APIs. It SHOULD avoid hidden local
state that changes platform behavior. Commands SHOULD be predictable,
scriptable, and safe for repeated use.

The CLI SHOULD prefer resource names, kinds, namespaces, and output formats that
match API concepts.

## Resource-Oriented Commands

The CLI SHOULD support:

- `version` for reporting the installed CLI package version without requiring
  platform storage.
- `health` for checking local platform store readiness or the API health
  endpoint.
- `apply` for declarative resource submission.
- `get` or `list` for resource retrieval.
- `describe` for resource summary, status, conditions, owners, and events.
- `delete` for resource deletion.
- `wait` for scriptable waiting on resource status conditions.
- `events` for event inspection.
- `trace` for Mission or AgentRun trace views.
- `approve` and `reject` for Approval decisions.
- `cancel` for declarative AgentRun cancellation.

Command names MAY vary, but equivalent capabilities SHOULD exist.

## Output

The CLI SHOULD support human-readable output and machine-readable output such as
YAML or JSON. Machine-readable output MUST preserve resource field names.

Human-readable output SHOULD emphasize phase, conditions, owner chain,
generation freshness, pending approvals, recent events, and artifact references.

For iterative AgentRuns, output SHOULD show execution-loop state, active
Decision summary, active ToolInvocation, pending Approval or input request,
budget usage, terminal reason, retryability, and partial Artifacts when
available.

## Describe

`describe` SHOULD aggregate useful information for one resource without changing
state. For a Mission, describe SHOULD show Workspace, Fleet, Agents, AgentRuns,
Context, Approvals, Artifacts, conditions, and recent events when available.

Describe output MUST NOT hide failure causes that are present in status or
events.

## Trace And Events

Trace commands SHOULD group correlated events and resources by Mission,
AgentRun, ToolInvocation, or correlation identifier. Tool-executing traces
SHOULD show Decision intent, Tool, operation, policy decision, execution result,
embedded Observation, and related Artifacts for each tool step.

Iterative AgentRun traces SHOULD show ExecutionFrame preparation, Decision
production and validation, Observation delivery, retries, budget updates, input
waiting, cancellation, timeout, budget exhaustion, finalization, and terminal
reason.

Event commands SHOULD support filtering by Workspace, kind, resource name, event
type, and correlation identifier.

Wait commands SHOULD support a timeout and SHOULD be safe to use in tutorials,
smoke tests, and automation. If a wait command can run reconciliation while
waiting, that behavior MUST be explicit.

The CLI cancellation command SHOULD expose intent equivalent to:

```text
platform cancel agentrun <name> -n <workspace>
```

Cancellation commands MUST NOT imply automatic rollback of side effects.

## Approval

Approval commands MUST require an explicit Approval identity. They SHOULD record
actor and reason. Approval commands MUST surface conflicts such as approving an
already rejected Approval.

`reject` SHOULD accept `--by`, `--reason`, and `--disposition terminate|continue`.
The default disposition MUST remain `terminate`. Operators MUST select
`continue` explicitly when the guarded action should be denied and its reason
returned to the execution loop for revision.

## Future Plugin Support

The CLI MAY support plugins. CLI plugins MUST NOT bypass API admission, policy,
or status ownership. Plugin commands SHOULD declare the API resources and
extension points they operate on.
